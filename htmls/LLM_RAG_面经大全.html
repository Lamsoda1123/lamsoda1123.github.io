RAG RAG是什么？与微调相比有什么优缺点？

RAG（Retrieval-Augmented Generation）是一种将信息检索技术与生成模型相结合的方法。其核心思想是将外部知识库中的相关信息检索出来，作为上下文输入给大语言模型，从而提高生成的准确性和可靠性。

优点：
  • 知识实时更新：无需重新训练模型，只需更新知识库即可获取最新信息
  • 减少幻觉：基于检索到的真实文档生成，降低模型编造事实的概率
  • 可解释性：可以追溯生成内容的来源，便于验证
  • 成本较低：相比微调，检索成本远低于训练成本
  • 领域适应性强：可以快速适应不同领域的知识需求
缺点：
  • 依赖检索质量：检索效果直接影响生成质量
  • 延迟较高：需要先检索再生成，增加响应时间
  • 知识碎片化：难以处理需要全局理解的问题
  • 上下文长度限制：检索到的文档可能超出模型上下文窗口
适用场景：知识库频繁更新、需要控制信息来源、减少幻觉的场景
LLM基础 Transformer的Attention机制是怎样的？QKV分别代表什么？

Transformer的注意力机制通过计算查询（Query）与所有键（Key）之间的相似度，得到注意力权重，然后用这些权重对值（Value）进行加权求和。

QKV的数学表达：

Attention(Q,K,V) = softmax(QK^T/√d_k)V

Q（Query）：查询向量，表示当前Token在查询什么信息；类比：我想了解什么
K（Key）：键向量，用于匹配查询；类比：我能提供什么信息
V（Value）：值向量，实际传递的信息；类比：实际要传递的内容

计算过程详解：

  1. 线性变换：输入X分别乘以三个权重矩阵得到Q、K、V
    Q = XW_Q, K = XW_K, V = XW_V
  2. 相似度计算：计算Q与所有K的点积，得到原始注意力分数
    scores = QK^T
  3. 缩放：除以√d_k（d_k是K的维度）防止梯度消失
    scores = scores/√d_k
  4. 掩码：对padding和未来词进行掩码处理
    scores = scores + mask
  5. Softmax：将分数转换为概率分布
    weights = softmax(scores)
  6. 加权求和：用权重对V加权求和得到输出
    output = weightsV
多头注意力：将QKV分成多个头，每个头学习不同的注意力模式，最后拼接起来。公式：MultiHead(Q,K,V) = Concat(head_1, ..., head_h)W_O
LLM训练 大模型的预训练、SFT和RLHF分别是什么？有什么区别？

大模型训练的三阶段：

1. 预训练（Pre-training）
  • 目标：学习语言基础能力和世界知识
  • 数据：海量无标注文本（如网页、书籍等）
  • 方法：下一个Token预测（自监督学习）
  • 计算量：最大，需要数千GPU天
  • 输出：基础模型（Base Model），具备通用能力但缺乏指令遵循能力
2. 监督微调（SFT - Supervised Fine-Tuning）
  • 目标：让模型学会遵循指令、进行对话
  • 数据：高质量指令数据集（问答对、对话等）
  • 方法：监督学习，最小化预测与标准答案的交叉熵
  • 计算量：较小，通常几小时到几天
  • 输出：指令微调模型，能遵循指令但可能存在有害输出
3. 人类反馈强化学习（RLHF - Reinforcement Learning from Human Feedback）
  • 目标：对齐人类价值观，减少有害输出
  • 数据：人类偏好数据（哪个回答更好）
  • 方法：
    • 训练奖励模型（RM）学习人类偏好
    • 使用PPO等强化学习算法优化策略
  • 计算量：中等，主要是推理成本
  • 输出：对齐模型（如ChatGPT），更安全、更符合人类偏好
关键区别：预训练学习"语言能力"，SFT学习"指令遵循"，RLHF学习"价值观对齐"
RAG优化 如何优化RAG的检索效果？有哪些常见的检索策略？

检索效果优化是多层次的系统工程：

1. 查询优化（Query Enhancement）
  • 查询扩展：使用同义词、相关词扩展原始查询
    原始：什么是注意力机制
    扩展：注意力机制、Attention Mechanism、self-attention、多头注意力
  • 查询重写：用LLM理解用户意图并优化查询
  • HyDE（Hypothetical Document Embedding）：让LLM先生成一个假设答案，再用答案的Embedding检索
  • 子查询分解：将复杂问题分解为多个子问题分别检索
2. 检索策略（Retrieval Strategies）
  • 密集检索（Dense Retrieval）：使用Embedding模型计算语义相似度
    cosine_similarity(query_embedding, doc_embedding)
  • 稀疏检索（Sparse Retrieval）：传统BM25等基于词频的方法
  • 混合检索（Hybrid Retrieval）：结合密集和稀疏检索的优点
    score = α * dense_score + (1-α) * sparse_score
  • 多向量检索：为每个文档生成多个向量（如针对不同类型的信息）
3. 索引优化（Indexing Optimization）
  • 分块策略：按段落、句子、固定长度等粒度切分文档
  • 重叠分块：相邻块之间保持重叠避免信息断裂
  • 层次化索引：建立多级索引（文档级、段落级、句子级）
  • 元数据增强：添加标题、关键词、时间等信息
4. 重排序（Reranking）
  • 交叉编码器（Cross-encoder）：用更大的模型精细计算query-doc相关性
  • 多阶段排序：粗排（快速召回）→ 精排（精细打分）
  • 多样性排序：避免返回重复内容
5. 反馈优化（Feedback Optimization）
  • 用户点击反馈：根据用户是否采纳调整检索权重
  • 在线学习：持续优化Embedding模型
  • A/B测试：对比不同检索策略的效果
实践建议：从简单方案开始（Embedding + 向量数据库），逐步添加查询优化和重排序，最后用反馈数据持续优化
微调技术 LoRA和QLoRA的原理是什么？相比全量微调有什么优势？
LoRA（Low-Rank Adaptation）核心思想：

不直接微调原始权重矩阵W，而是添加一个低秩的增量ΔW：

W' = W + ΔW = W + BA

其中：
B ∈ ℝ^(d×r), A ∈ ℝ^(r×k)
r << min(d,k) 是低秩的秩数

为什么是低秩的？

  • 模型微调时权重更新ΔW通常是低秩的（大部分信息可以用少数几个方向表示）
  • 通过低秩分解大幅减少参数量：从d×k降到(d+k)×r
  • 只训练A和B，原模型权重W冻结
QLoRA（Quantized LoRA）改进：

在LoRA基础上增加量化，进一步降低显存：

  • 4-bit NormalFloat量化：将原始权重W量化为4-bit，大幅减少显存占用
  • 双重量化：对量化常数进行二次量化
  • 分页优化器：使用NVIDIA统一内存管理显存
  • LoRA层保持16-bit：确保训练稳定性
相比全量微调的优势：
维度 全量微调 LoRA/QLoRA
显存占用 极高（需存储梯度、优化器状态） 极低（冻结大部分参数）
训练速度 慢（参数更新量大） 快（仅更新少量参数）
存储空间 每个微调版本需存储完整模型 只需存储小的adapter（几十MB）
效果 理论上限最高 接近全量微调效果
任务切换 需要重新加载整个模型 只需切换adapter
实践建议：在消费级GPU（如RTX 3090/4090）上微调7B/13B模型时，QLoRA是最佳选择。秩r通常取8-64，α通常取r的2倍

  • 时间加权：对新闻类查询偏好近期文档

  • 相关性过滤：设置阈值剔除低相关性文档（阈值0.3）

  • 自一致性检查：多次生成同一问题，检查答案是否一致

混元大模型的数据处理流水线（基于公开资料）：

1. 数据采集阶段

  • 多源数据融合：网页、书籍、新闻、学术论文、代码、对话
  • 总量控制：原始数据约5T tokens，经过筛选保留约2T高质量tokens
  • 领域平衡：按照预设比例采样（中文:英文=7:3，技术:通用=4:6）

2. 数据清洗管道（4层过滤）

  • 第一层：基础过滤
    • 去重：MinHash + LSH算法，相似度>90%的文档去重
    • 长度过滤：剔除过短（<100字符）和过长（>10万字符）的文档
    • 格式清洗：去除HTML标签、特殊字符等
  • 第二层：质量打分
    • 使用BERT-base训练质量分类器，评分0-1
    • 保留分数>0.6的文档
    • 评分维度：语法正确性、逻辑连贯性、信息密度
  • 第三层：安全过滤
    • 敏感词过滤（基于腾讯安全词库）
    • 毒性检测（Toxicity Detection）
    • 个人隐私信息过滤（身份证号、手机号等）
  • 第四层：人工抽检
    • 随机抽取1%数据人工审核
    • 抽检指标：准确率>99%，召回率>95%
    • 不合格则回溯调整自动流程

3. 数据增强策略

  • 代码数据：函数-注释对、代码-文档对，占比约15%
  • 数学数据：数学问题-解题步骤对，占比约5%
  • 对话数据：从公开对话数据中清洗整理，占比约10%
  • 多语言对齐：中英文平行语料，提升跨语言能力

4. 数据存储与管理

  • 版本控制：每批数据都有唯一ID和元数据
  • 血缘追踪：记录数据来源、处理步骤、质量评分
  • 动态更新：每月增量更新5-10%数据

广告/推荐 腾讯广告大模型如何处理冷启动问题？
腾讯广告大模型冷启动问题的多维度解决方案：

1. 元学习（Meta-Learning）策略

  • MAML算法应用：在训练阶段模拟冷启动任务
    训练任务：每个广告主的广告视为一个独立任务
  • 快速适配：用少量样本（10-50次曝光）快速调整上层参数
  • 效果：冷启动CTR预估准确率提升25%

2. 迁移学习机制

  • 广告主画像编码：将行业、预算、历史表现等编码为向量
  • 相似广告主迁移：找到相似广告主，复用其模型参数作为初始化
  • 层次化迁移：
    • 行业级参数（跨广告主共享）
    • 广告主级参数（特定广告主）
    • 广告级参数（特定广告）

3. 多塔结构（Multi-Tower）设计

  • 共享塔：所有数据训练，学习通用特征（年龄、性别、兴趣等）
  • 私有塔：每个广告主独立训练，学习特定特征
  • 动态权重：根据数据量自动调整共享塔和私有塔的权重
    新广告：共享权重=0.8, 私有权重=0.2
    成熟广告：共享权重=0.3, 私有权重=0.7

4. 探索与利用（Exploration）平衡

  • ε-greedy策略：冷启动阶段提高探索率到30%
  • UCB算法：对不确定性高的广告增加曝光机会
  • Thompson Sampling：基于贝叶斯方法平衡探索与利用

5. 辅助信息利用

  • 创意特征：分析广告图片/文案的语义特征与历史相似广告
  • 落地页分析：解析广告落地页内容，提取关键词和类别
  • 自动标签：用CV/NLP模型自动打上内容标签，匹配目标用户

推荐回答思路：

1. Query理解增强（Query Understanding）
  • 意图识别：用分类模型识别Query意图类型
    示例："苹果" → 意图分类：科技(0.6) vs 水果(0.3) vs 公司(0.1)

    用户之前问"iPhone 15"，当前问"价格" → 合并为"iPhone 15价格"

  • 隐性扩展（Implicit Expansion）：
    • 同义词扩展：用Word2Vec/ESimCSE找相似词
    • 上下位词扩展："手机"→"iPhone"、"华为"、"小米"

    • HyDE：让LLM生成假设答案，用答案去检索
    • Query2Doc：LLM生成伪文档，用伪文档检索
    • 多跳扩展：先检索一批相关文档，再从文档中提取关键词二次检索

  • 多级索引：
    • 粗粒度：文档级索引，快速定位候选文档集
    • 细粒度：段落级/句子级索引，精确定位答案
  • 稀疏+密集混合检索：
    • BM25：对关键词匹配敏感
    • Embedding：对语义相似敏感
    • 加权融合：score = 0.3*bm25_score + 0.7*embedding_score
  • 文档权重调整：根据文档质量、时效性、权威性加权
4. 生成侧验证（Generation Verification）
  • Recall and Verify：
    • 召回Top-K文档后，用LLM判断相关性
    • 筛除不相关文档，减少干扰信息
    • Prompt: "判断以下文档是否能回答Query，只输出是/否"
  • 自一致性检验：
    • 多次生成答案，检查是否一致
    • 如果不一致，说明检索效果不佳

  • 使用NumPy实现矩阵运算（题目说不能用tensor，没说不能用np）

import numpy as np

class LinearRegression:
    def __init__(self, input_dim, learning_rate=0.01):
        # 初始化权重和偏置
        self.w = np.zeros(input_dim)
        self.b = 0
        self.learning_rate = learning_rate

    def forward(self, X):
        """前向传播: y = X @ w + b"""
        return np.dot(X, self.w) + self.b

    def backward(self, X, y_pred, y_true):
        """
        反向传播: 计算梯度
        参数：
            X: (batch_size, input_dim)
            y_pred: (batch_size,)
            y_true: (batch_size,)
        """
        batch_size = X.shape[0]

        # 损失函数: MSE = 1/n * Σ(y_pred - y_true)²
        # 对w的梯度: dL/dw = 2/n * Xᵀ(y_pred - y_true)
        dw = (2 / batch_size) * np.dot(X.T, y_pred - y_true)

        # 对b的梯度: dL/db = 2/n * Σ(y_pred - y_true)
        db = (2 / batch_size) * np.sum(y_pred - y_true)

        return dw, db

    def update_params(self, dw, db):
        """梯度下降参数更新"""
        self.w -= self.learning_rate * dw
        self.b -= self.learning_rate * db

    def train(self, X, y, epochs=100):
        """训练循环"""
        for epoch in range(epochs):
            # 前向传播
            y_pred = self.forward(X)

            # 计算损失
            loss = np.mean((y_pred - y) ** 2)

            # 反向传播
            dw, db = self.backward(X, y_pred, y)

            # 参数更新
            self.update_params(dw, db)

            if epoch % 10 == 0:
                print(f"Epoch {epoch}, Loss: {loss:.4f}")

算法/概率 4x4棋盘上的马从(0,0)出发，首次到达四个角的期望步数是多少？
这是一道经典的马尔可夫链问题

问题分析：

  • 棋盘：4x4 = 16个格子，坐标(0,0)到(3,3)
  • 起点：(0,0)，终点：四个角 (0,0)、(0,3)、(3,0)、(3,3)
  • 马的走法："马走日"，日字形的8个方向
  • 目标：计算首次到达任意一个角的期望步数

解法：

  1. 建立状态转移方程：对每个非终点状态，记录到达各个角的期望步数
  2. 期望公式：E[i] = 1 + Σ(P(i→j) * E[j])，其中j是i可以到达的状态
  3. 吸收态：四个角的期望为0（已经到达）
  4. 解线性方程组：构建16个方程，求解线性系统
import numpy as np
from scipy.linalg import solve

def knight_expectation():
    board_size = 4
    moves = [(2, 1), (2, -1), (-2, 1), (-2, -1),
             (1, 2), (1, -2), (-1, 2), (-1, -2)]

    # 将2D坐标转换为1D索引
    def idx(x, y):
        return x * board_size + y

    # 判断是否为终点（四个角）
    def is_corner(x, y):
        return (x == 0 or x == 3) and (y == 0 or y == 3)

    n_states = board_size * board_size
    A = np.zeros((n_states, n_states))
    b = np.zeros(n_states)

    for x in range(board_size):
        for y in range(board_size):
            i = idx(x, y)

            if is_corner(x, y):
                # 吸收态: E = 0
                A[i, i] = 1
            else:
                # 建立方程: E[i] = 1 + Σ(E[next]) / 可选步数
                A[i, i] = 1
                b[i] = 1

                valid_moves = []
                for dx, dy in moves:
                    nx, ny = x + dx, y + dy
                    if 0 <= nx < board_size and 0 <= ny < board_size:
                        valid_moves.append((nx, ny))

                n_moves = len(valid_moves)
                for nx, ny in valid_moves:
                    j = idx(nx, ny)
                    A[i, j] = -1 / n_moves

    # 解线性方程组
    E = solve(A, b)

    # 结果: (0,0)的起点期望
    print(f"从(0,0)到角的期望步数: {E[idx(0,0)]:.2f}")
    return E[idx(0,0)]

Reward Model Reward Model的训练策略有哪些？On-policy vs Off-policy的区别是什么？
二面重点考察对RLHF的理解

Reward Model训练策略对比：

1. On-Policy策略（如DPO Direct Preference Optimization）
  • 训练流程：
    • 用当前模型生成response
    • 人工标注偏好（哪个response更好）
    • 根据偏好label更新RM模型
  • 优点：
    • Reward与当前模型严格对齐
    • 不需要显式训练Reward Model
    • 实现更简单稳定（无reward hacking）
  • 缺点：
    • 数据收集成本高：每次更新都需要重新采样
    • 训练不稳定性：策略改变快导致reward估计偏差
2. Off-Policy策略（标准RLHF流程）
  • 训练流程：
    • 提前收集一批response对并标注
    • 用所有历史数据训练Reward Model
    • 策略更新后，Reward Model不变，用于RL
  • 优点：
    • 样本复用率高：历史数据可重复利用
    • 训练稳定：Reward模型固定，计算准确
    • 适合监督信号稀疏的场景
  • 缺点：
    • Distribution Shift：策略改变后，Reward可能不准确
    • Two-stage训练复杂，需要额外reward训练
    • 可能出现reward hacking问题
3. 混合策略（Hybrid）
  • 定期更新Reward Model：每隔N步policy更新后，用On-policy数据重训RM
  • 重要性采样（Importance Sampling）：
    • 调整Off-policy数据的权重
    • 降低与当前策略差异大的样本权重
实践建议：
  • 学术研究：DPO（On-policy）更简单，效果好
  • 工业场景：Off-policy/hybrid更实用，样本成本更低
  • Kimi内部同时支持两种方式，根据任务切换
多模态/Vision Vision特征处理有哪些形式？Encoder+Adapter的模式如何设计？
二面考察多模态能力（Kimi正在做多模态方向）
1. Vision特征处理形式
  • Pixel-level Tokenization（早期LLaVA方案）：
    • 用CLIP ViT的每个patch输出作为token
    • 576个patch × 1024维 = 大量token
    • 问题：占用太多context length，计算开销大
  • Grid-level Tokenization（无投影）：
    • 将图像分割为固定网格（如12×12）
    • 每个网格作为一个visual token
    • 问题：丢失细粒度信息
  • Hybrid Tokenization（当前主流）：
    • 全局特征：1-2个token表示整图
    • 局部特征：重要区域的patch token（用saliency检测）
    • 动态token数量：根据图像复杂度调整
  • Embedding-space Projection（LLaVA-v1.5）：
    • CLIP嵌入 → Linear层 → LLM embedding空间
    • 10-20个visual token就够
    • 优势：token数量少，接入简单
2. Encoder+Adapter架构设计
  • Encoder层（预训练）:
    • CLIP ViT-Large (336px)或EVA-02
    • 冻结权重，避免catastrophic forgetting
    • 输出: [num_patches, hidden_dim]
  • Adapter层（可训练）:
    • Q-Former（BLIP-2）：学习最相关的visual token
      • 固定数量的learnable queries（如32个）
      • Cross-attention从ViT输出中提取信息
    • Linear Projection：最简单的方式
      • [576, 1024] → [32, 4096] 投影
      • 适配LLM的embedding维度
    • MLP Adapter：通用多层感知机
3. Alignment策略
  • Full Fine-tuning: Encoder + Adapter + LLM全量训练
    • 成本极高，需要处理catastrophic forgetting
  • LoRA on LLM: 只微调LLM的attention层（当前最优）
    • 冻结Encoder和大部分LLM
    • 在Q,V投影矩阵上加LoRA（rank=64）
    • 效果：保持LLM语言能力 + 学习视觉理解
Kimi当前方案：Embedding-space Projection + LoRA，token数20，batch推理速度提升3x

MoE架构 请比较一下MoE、稠密模型和MoA（Mixture of Agents）的特点和适用场景，DeepSeek为什么选择MoE？
三种架构的详细对比：
1. 稠密模型（Dense Model）
  • 特点：所有参数都参与计算，结构简单，易于优化
    • 训练稳定，收敛性好
    • 推理速度快（无需路由计算）
    • 模型能力上限受限
  • 适用场景：小规模模型（<30B），对延迟敏感的场景
2. MoE（Mixture of Experts）
  • 特点：稀疏激活，每次只激活部分专家
    • 参数效率高：8x7B模型实际激活14B参数
    • 推理开销小：保持小模型推理速度
    • 扩展性好：专家数量可以按需扩展
    • 负载均衡挑战：需要精心设计路由机制
  • DeepSeek的选择原因：
    • 推理效率：激活参数少，适合部署在有限资源环境下
    • 性能扩展：总参数量大，模型能力强
    • 成本优势：预训练成本是稠密模型的1/3
    • 路由优化：DeepSeek-MoE使用gated routing，负载均衡效果好
3. MoA（Mixture of Agents）
  • 特点：多个独立模型协作，通过投票/加权方式决策
    • 灵活性高：可以组合不同类型的模型
    • 推理成本高：需要运行多个模型
    • 协作复杂：需要解决模型间的一致性问题
  • 应用场景：集成多个专业模型（数学+代码+写作）
实用建议：
• 资源充足+追求效果：MoE（推荐DeepSeek-V2架构）
• 推理延迟敏感：稠密模型
• 复数专业模型：MoA（最佳集成方案）

  • 注意力分组：按主题/段落分组注意力计算
    • Chunk-wise local attention
    • Dilated attention（跳过部分token）
2. 位置编码改进
  • NTK-aware interpolation：
    • 高频特征保持更多的周期，低频拉长
    • 外推效果好，四倍缩放后困惑度仅增加5%
    • 推理时不需重新训练
  • YaRN：引入温度系数修正注意力分布
    • 超过128k后效果优于NTK
    • 最大限度保留短上下文性能
3. KV缓存管理
  • 分层存储：
    • GPU显存存最近4k token
    • CPU内存存中间32k token
    • 硬盘存剩余的92k token
    • 按需加载，延迟增加<10%
  • 压缩技术：用小的embedding替代完整token
    • 平均压缩率8x
    • 精度损失<2%
